# Ralph Golden-Path Review

Archived note: this document describes the pre-story-task Ralph architecture and intentionally references legacy commands such as `ralph-prd.sh`, `ralph-prime.sh`, and `ralph-archive.sh`. Those commands are no longer part of the active framework and are kept here only as historical context.

## Findings

### 1. The implemented flow mostly matches the intended artifact policy

The current framework is largely aligned with the desired rule:

- Keep durable specs with the codebase
- Keep runtime loop state transient
- Preserve runtime evidence in archive output

What is true today:

- Standalone PRD markdown generated by [`ralph-prd.sh`](../ralph-prd.sh) is committed when created.
- Epic PRD markdown generated from `promptContext` by [`ralph-prime.sh`](../ralph-prime.sh) is committed when created.
- Runtime state such as `scripts/ralph/prd.json`, `scripts/ralph/progress.txt`, `.iteration-log*`, `.iteration-handoff*`, and `.playwright-cli` remains transient.
- Runtime artifacts are preserved by [`ralph-archive.sh`](../ralph-archive.sh) under `scripts/ralph/tasks/archive/`.

This is the strongest match to the intended gold standard that the framework has had so far.

### 2. The roadmap planning gap is now addressed with a first-class planner

The framework now has strong support for:

- sprint containers
- epic backlogs
- roadmap decomposition into sprint backlogs
- dependency-aware epic ordering
- PRD generation from `promptContext`
- loop execution
- archive and merge closeout

[`ralph-roadmap.sh`](../ralph-roadmap.sh) now fills the missing top layer by:

- taking a broad roadmap/future-state description
- decomposing that into multiple sprints
- emitting a complete, ordered sprint/epic plan
- assigning sprint-safe epic effort scores
- enforcing sprint capacity target/ceiling during planning
- keeping explicit dependencies sprint-local while using sprint order for cross-sprint sequencing
- preserving a durable roadmap source that can be refined later and reconciled back into sprint backlogs
- distinguishing roadmap-managed epics from local ad hoc refinements so roadmap updates do not trample local backlog work

That closes the largest structural golden-path gap without changing the existing loop primitives.

### 3. Spec tracking is now coherent, and scoped execution rules now have light runtime enforcement

The system now behaves correctly in the happy path:

- specs are durable
- runtime state is transient
- archived runtime evidence is preserved separately

The largest scoped-execution gap is now partially closed in [`ralph.sh`](../ralph.sh), and the core behavior is now backed by direct regression coverage.

Examples:

- when PRD/source text clearly says source changes are limited to named files, `ralph.sh` now validates that iteration commits stay inside that explicit source scope
- verification/test-file expansion is allowed outside that scope, but only for verification-oriented paths
- regression coverage now proves both the allowed verification-expansion path and the blocked out-of-scope source-edit path
- story decomposition quality is mostly prompt-level, not schema-enforced beyond basic structure checks

That is a good tradeoff: it hardens the highest-value scoped-work rule without adding a heavy policy engine. Some correctness still depends on model compliance, but less than before.

### 4. Worst-case smoke is now stronger because it validates runtime behavior

The worst-case smoke test in `scripts/smoke/e2e-worst-case-ui.sh` has been valuable because it caught real behavior problems:

- scope drift into helper scripts
- verification catch-22 behavior
- stale-lock retry behavior

It now also validates the UI contract at runtime instead of depending on exact implementation spelling for state/title handling.

That is a better fit for this harness:

- it still enforces strict file-scope discipline
- it still verifies the required UI contract
- it is less likely to false-fail on equivalent implementations

### 5. Smoke-harness lock handling is now safer, but only in test automation

The stale workflow lock issue was fixed in:

- `scripts/smoke/e2e-sanity.sh`
- `scripts/smoke/e2e-worst-case-ui.sh`

This was the right low-risk choice.

Current reality:

- smoke retries can clear a provably stale `.workflow-lock` in disposable repos
- core Ralph lock semantics in [`ralph.sh`](./ralph.sh) and [`ralph-prime.sh`](./ralph-prime.sh) were not weakened

That matches the intended safety model, but it also means:

- if a similar stale-lock issue appears in a real project run, it will still need a separate core fix

So this is resolved for smoke reliability, not universally for all future operator scenarios.

### 6. The leaner loop prompt is better, but prompt trimming can still distort benchmark semantics

Recent prompt trimming improved token efficiency in the always-paid loop prompt.

Current loop prompt strengths:

- clearer scoped-verification rule
- less repeated wording
- lower ongoing token cost

Observed risk:

- trimming benchmark-specific wording too aggressively changed the intended meaning of the worst-case scenario
- this reduced token use, but at the cost of benchmark correctness

That means the loop prompt can be kept lean, but scenario prompts used for measurement need to preserve exact semantic distinctions.

## Golden Path vs. Reality

### Intended gold standard

The intended process appears to be:

1. Install framework into a clean repo
2. Create roadmap direction
3. Map roadmap into ordered sprints and epics
4. Generate durable PRD specs
5. Prime transient runtime state from those specs
6. Execute one story per clean-context loop iteration
7. Archive runtime artifacts
8. Merge code/spec history forward
9. Repeat for next epic, then next sprint

### How reality matches

From step 4 onward, reality is strong.

- PRD markdown is durable and committed.
- Runtime state is transient.
- The loop works one story at a time.
- Completion detection is good.
- Archive and merge phases are well separated.
- Sprint closeout exists and is structurally sound.

From steps 1 to 3, reality is now much closer to the intended path.

- installation is solid
- sprint creation is solid
- epic backlog management is solid
- roadmap-to-sprint decomposition now has a first-class planner

So the real answer is:

- **execution path**: largely matches the gold path
- **upfront roadmap planning path**: now materially matches the gold path

## Simplified Golden Path

The current repo now supports a much simpler operator story than earlier versions:

1. Install Ralph into the target repo
2. Run `doctor.sh`
3. Choose one planning path:
4. Standalone: `ralph-prd.sh`
5. Sprint: `ralph-roadmap.sh`
6. Run `ralph.sh`
7. Run `ralph-commit.sh` after each completed PRD or epic
8. In sprint mode, finish with `ralph-sprint-commit.sh`

That is the documentation shape the repo should optimize around. Everything else is advanced control surface.

## Process Outline

### Fresh baseline to planning-ready

1. Run [`install.sh`](../install.sh)
2. Run [`doctor.sh`](../doctor.sh)
3. Generate roadmap/sprint backlogs with [`ralph-roadmap.sh`](../ralph-roadmap.sh)
4. Review or activate sprint with [`ralph-sprint.sh`](../ralph-sprint.sh)
5. Refine epics manually with [`ralph-epic.sh`](../ralph-epic.sh) only when needed

### Planning phase

Standalone path:

1. Run [`ralph-prd.sh`](../ralph-prd.sh)
2. Codex evaluates the PRD-generation prompt
3. Codex uses `prd` then `ralph` skill
4. PRD markdown is written and committed
5. `prd.json` is written as transient execution input
6. `.active-prd` is written in standalone mode

Epic path:

1. Normal path: run [`ralph.sh`](../ralph.sh), which auto-calls [`ralph-prime.sh`](../ralph-prime.sh) when needed
2. Next eligible epic is selected from `epics.json`-style backlog
3. If needed, markdown PRD is generated from `promptContext`
4. Markdown PRD is converted to transient `prd.json`
5. Epic is marked active
6. Generated epic markdown and active epic status are committed
7. Local runtime state is reset for clean loop startup

### Loop prep

1. Run [`ralph.sh`](../ralph.sh)
2. Lock is acquired
3. Transient files are verified untracked
4. Backlog inputs are checked clean
5. PRD is validated or primed
6. Feature branch from `branchName` is created/checked out
7. Prior run artifacts are checked for required archive
8. Progress log is initialized if needed

### Loop execution

Per iteration:

1. Render [`prompt.md`](../prompt.md) with live PRD/progress paths
2. Inject local prompt sections from [`prompt.local.md`](../prompt.local.md) if present
3. Run Codex in fresh context
4. Require one story only
5. Require targeted verification and browser evidence when needed
6. Commit story work
7. Update transient PRD/progress state
8. Check stable completion

### Loop closeout

Per completed PRD:

1. Run [`ralph-commit.sh`](../ralph-commit.sh)
2. Validate all stories pass
3. Run [`ralph-archive.sh`](../ralph-archive.sh)
4. Verify archive correctness
5. Merge feature branch into correct target
6. Update epic status if epic mode
7. Clear local active run markers

Per completed sprint:

1. Run [`ralph-sprint-commit.sh`](../ralph-sprint-commit.sh)
2. Validate all sprint epics terminal
3. Archive sprint metadata
4. Merge sprint branch to `main`/`master`
5. Clear active sprint state

## Recommendations

### 1. Keep roadmap planning thin and artifact-first

The new roadmap layer is the right shape:

- durable roadmap outputs
- durable roadmap source
- sprint/epic metadata stored in existing structures
- no runtime-state coupling
- no change to loop or closeout primitives

That should remain the design rule for future roadmap-planning iterations.

### 2. Keep roadmap refinement additive by default

Refinement should flow from the durable roadmap source into open and future sprint backlogs.

Recommended default:

- update open and future work in place
- preserve active and completed work unless a low-churn reopen is clearly safer
- prefer new follow-up epics or new sprints over reopening closed sprints when the refinement would otherwise create broad churn

### 3. Keep enforcing durable-spec / transient-runtime separation

Current policy is good and should stay:

- commit markdown specs
- keep `prd.json` and `progress.txt` transient
- archive runtime evidence only

This part should not be loosened.

### 4. Keep light validation narrow and focused on explicit scope contracts

What is now covered:

- explicitly file-scoped source changes should remain inside named implementation files
- verification expansion outside that source scope should remain verification-only

What should stay prompt-level for now:

- broad story-shaping quality
- nuanced decomposition judgment
- non-explicit scope interpretation

This remains the right design rule: validate only low-ambiguity, high-value constraints.

### 5. Keep smoke assertions behavior-led when equivalent implementations are acceptable

The recent worst-case smoke update is the right pattern:

- keep strict scope checks
- keep runtime contract checks
- avoid asserting exact source spelling unless the spelling itself is part of the requirement

### 6. Continue monitoring real-world lock behavior separately from smoke behavior

Current smoke fix is appropriate.

Recommendation:

- keep watching for stale-lock failures in non-smoke project runs
- only then decide whether core lock behavior needs PID-aware or stale-owner logic

### 7. Keep the loop prompt lean, but avoid over-compressing benchmark scenario prompts

Best practice:

- trim the always-paid loop prompt aggressively
- keep scenario definitions exact when they are part of regression measurement

This preserves both efficiency and trustworthy benchmarks.

## Bottom Line

The framework now matches the intended artifact policy well:

- specs with code
- runtime state transient
- runtime evidence archived

The main gap is not in the first planning layer anymore.

The main remaining issues are lighter-weight and more tactical:

- some higher-level behavior rules still intentionally remain prompt-led rather than validator-enforced
- real-world lock behavior should still be observed separately from smoke-only behavior
- benchmark scenario prompts still need careful semantic preservation when trimmed

That means Ralph is now strongest as:

- a roadmap-to-sprint planning and PRD-to-execution framework
- with its main remaining risks in test-harness sharp edges and selective prompt-led policy areas, not core workflow shape

## Status Summary

### Resolved

- Artifact policy now matches the intended model: durable specs are committed, runtime state stays transient, and runtime evidence is archived.
- Roadmap-to-sprint planning now has a first-class implementation in [`ralph-roadmap.sh`](../ralph-roadmap.sh).
- Explicit file-scoped execution rules now have light runtime enforcement in [`ralph.sh`](../ralph.sh), with regression coverage for both allowed and blocked cases.
- Worst-case smoke now validates runtime UI behavior instead of relying on brittle implementation spelling.
- Smoke retry handling now safely clears only provably stale locks in disposable smoke repos.

### Monitor

- Real-world stale-lock behavior outside smoke repos should still be observed before changing core lock semantics.
- Benchmark scenario prompts should still be watched carefully during prompt trimming so semantic drift does not invalidate measurements.

### Intentional Prompt-Led Areas

- Broad story decomposition quality remains prompt-led.
- Nuanced planning judgment and non-explicit scope interpretation remain prompt-led.
- These areas are intentionally not validator-enforced unless repeated low-ambiguity failures justify a narrow guardrail.

## Updated Priority Order

1. Continue monitoring real-world stale-lock behavior before changing core lock semantics.
2. Keep benchmark scenario prompts semantically exact while continuing to trim the always-paid loop prompt.
3. Keep the explicit-scope validator narrow and extend it only if repeated real failures show a clear low-ambiguity pattern.
